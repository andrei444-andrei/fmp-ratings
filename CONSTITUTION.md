# Конституция продукта

Главный документ для управления архитектурой продукта. Правила здесь —
обязательные и неизменные по именам/контрактам; любой чат, агент или
разработчик начинает работу отсюда. Технические детали реализации живут в коде
и в `README.md`, но принципы и закреплённые имена — здесь.

---

## 1. Схема БД — без ручных шагов

- **Самопровижининг и идемпотентность.** На чистой БД приложение поднимается
  само, без ручного SQL/миграций: либо `CREATE TABLE IF NOT EXISTS` при первом
  обращении (кэш и вспомогательные таблицы), либо миграции, запускаемые
  автоматически на деплое/старте. Ручной шаг = баг процесса.
- **Один источник правды на таблицу + мягкий сбой.** Не дублируй определение
  (или ORM-схема, или ленивый `CREATE` — не оба). Провал создания не валит
  приложение (`try/catch` + лог), но помни: read-only токен БД ломает запись
  молча — проверяй права.
- **Движок БД.** Стек проекта — **Turso/libSQL (SQLite)** через Drizzle ORM (а не
  Neon/Postgres, как изредка мелькало в черновиках). Один драйвер, одна схема.
- **Обязательное `created_at`** во всех таблицах (ISO-время / timestamptz, default
  «сейчас»): без него нельзя безопасно вычищать тестовые данные по окну работы
  агента (§5). Нет `created_at` — таблица не годится.

## 2. Логи — единое место, известное любому чату

- **Один сток, одна форма.** Все ошибки (бэкенд + клиент) пишутся в одну таблицу
  БД **`app_errors`** (на serverless только БД — файл эфемерен) в едином формате:
  `ts`, `level`, `source` (`server` | `client`), `route`, `message`, `stack`,
  `build`, `user_agent`, `meta`.
- **Фиксированная точка чтения.** Дамп последних ошибок отдаёт
  **`GET /api/admin/errors?limit=N`**. Эти два имени (`app_errors` /
  `/api/admin/errors`) закреплены здесь и неизменны — любой агент/разработчик
  начинает диагностику отсюда, не спрашивая «а где логи?».
- **Правило для сессии:** при разборе бага сначала смотри `app_errors` /
  `/api/admin/errors`, чини по стектрейсу; пользователю — понятное сообщение +
  `error_id` для поиска в логе.

## 4. Отладочная страница БД (в каждом проекте)

- С самого старта — **защищённая** (под токеном/авторизацией, **никогда не
  публичная**) служебная страница **`/admin`**, показывающая актуальную структуру
  БД: таблицы и колонки с описанием на русском (что значит каждая таблица/поле).
  Описания берутся из единого реестра/комментариев схемы и держатся в синхроне с
  реальными таблицами.
- **Просмотр значений:** выбрал таблицу → постранично смотришь строки (read-only
  по умолчанию; произвольный SQL и мутации — отдельно и тоже под защитой). Туда
  же вешается дамп `app_errors`.
- **Цель:** любой чат/агент и ты сам инспектируете данные и схему без ручного SQL
  и без консоли БД — диагностика начинается отсюда.
- **Важно:** эта страница раскрывает структуру и данные, поэтому доступ к ней —
  обязательно за авторизацией; в публичный доступ не выносить.

## 5. Сначала протестируй — потом докладывай

- **«Собирается» ≠ «работает».** Никогда не пиши пользователю «готово/работает»
  на основании одних лишь `tsc`/`build`. Сначала запусти и понаблюдай реальное
  поведение на сценарии из задачи + на крайних случаях.
- **Тестируй в реальных условиях,** если они доступны (это обязательная часть
  задачи, не опция): есть ключи (staging) — подними приложение и побей реальные
  эндпоинты; UI — прогони в браузере (Playwright/скриншот); по возможности —
  против preview-деплоя (настоящий serverless: таймауты, лимиты).
- **E2E в браузере — обязательная часть «готово» для UI** (не «по возможности»).
  Реальный браузер (Playwright): основной сценарий + мобильный вьюпорт,
  `npm run test:e2e`. Скриншота недостаточно — нужен зелёный e2e.
- **Что проверить нельзя — отметь честно.** Нет ключа/браузера/прод-условий → не
  пиши «работает», а: «сделано, собирается; НЕ проверено вживую: <что>;
  вероятный риск: <где>» — и попроси доступ для теста или подтверждение.
- **Мини-чек перед каждым «готово»:** воспроизвёл основной сценарий? проверил
  edge-кейсы и пустые/битые данные? фича не падает без ключей (graceful)? Хоть
  один «нет» — значит ещё не готово.

## 6. Рыночные данные — провайдеры и канонические ключи

- **Два провайдера, разделение ответственности (закреплено):**
  - **Цены (дневные EOD)** — **EODHD ОСНОВНОЙ** (`EODHD_API_KEY`):
    скорректированные цены (`adjusted_close` → корректные сплиты/дивиденды,
    широкое гео для иностранных бирж). **FMP — ДОБОР/ФОЛБЭК** (`FMP_API_KEY`):
    заполняет пробелы по символам/формам, которых нет в EODHD.
  - **Фундаментал** — **FMP** (`FMP_API_KEY`): скринер компаний по
    стране/сектору/капитализации, профиль, earnings/EPS, дивиденды, рыночная
    капитализация, аналитические грейды. (EODHD для фундаментала не используем.)
- **Канонические имена env (неизменны):** `EODHD_API_KEY`, `FMP_API_KEY`.
- **Единый слой доступа, кэш-первым.** Фичи НЕ ходят в провайдера напрямую — только
  через общий слой (`getPrices` / `getFundamentals` и т.п.) с кэшем в БД
  (`prices`/`price_meta`, `fundamentals`, `dividends`, `signals_universe`) и
  TTL/эпохой инвалидации. Предохранители (don't-downgrade + эпоха) не дают
  короткому/битому ответу провайдера ухудшить рабочий кэш.
- **Graceful без ключей.** Нет ключей (e2e/демо) → детерминированная синтетика и
  то, что уже в кэше; приложение и метрики не падают (см. §5).

## 7. Без тупых approval-промптов — allow-лист в репозитории

- **Цель.** Рутинная диагностика не должна дёргать пользователя диалогом
  «Allow Claude to use … (Allow once / Always allow)». Read-only инструменты,
  которыми смотрят логи/деплои/ошибки, заранее разрешены — во всех чатах проекта.
- **Где это живёт.** Список разрешений — в **`.claude/settings.json`**
  (`permissions.allow`), который **коммитится в репозиторий** (в отличие от
  `settings.local.json` — он per-user и в `.gitignore`, поэтому в новых чатах не
  действует). Только коммит даёт «само применяется во всех чатах проекта».
- **Что разрешено.** Read-only/диагностические инструменты Vercel
  (`get_runtime_errors`, `get_runtime_logs`, `get_deployment*`, `list_*`,
  `get_project`, `web_fetch_vercel_url`, `get_access_to_vercel_url`,
  `search_vercel_documentation`). Появился новый безопасный read-only инструмент,
  который зря спрашивает доступ → добавь его в этот же allow-лист и закоммить.
- **Граница.** В allow-лист идут только НЕразрушающие операции. Изменяющие/
  внешне-видимые действия (деплой, покупка доменов, мутации) — НЕ авто-разрешать.
- **Поведение.** Где возможно, выбирай непрашивающий путь (например, читать
  ошибки прямым запросом к `app_errors` или `GET /api/admin/errors`), а не
  approval-gated инструмент.

## 9. Доставка на прод — автоматически через `main`, из любого чата

- **Прод привязан к `main`.** Production-ветка Vercel-проекта `fmp-ratings` (§8) —
  `main`. Любой коммит, попавший в `main`, **автоматически деплоится на прод**
  (https://fmp-ratings.vercel.app) через Git-интеграцию Vercel — без `vercel --prod`,
  без отдельной команды деплоя, из любого чата проекта. Это и есть «залить на прод».
- **«Готово» = в проде, а не в ветке.** Задача не закрыта, пока изменения не в
  `main`. Каждый чат доводит цикл до конца самостоятельно: push → PR (обычный,
  не draft) → зелёный обязательный CI → squash-merge в `main` → авто-деплой на прод.
  Висящая ветка/PR без мержа = недоставленная работа.
- **Не деплой руками.** Прод-деплой — следствие мержа в `main`, а не отдельное
  внешне-видимое действие под авторизацию (ср. §7). Не дёргай `deploy_to_vercel` /
  `vercel --prod` вручную: это плодит деплои мимо `main` и рассинхрон. Источник
  правды для прода — содержимое `main`.
- **Проверка факта (§5).** После мержа убедись, что прод реально обновился: запрос
  к https://fmp-ratings.vercel.app по затронутому маршруту/поведению. Не обновился —
  инцидент процесса: проверь, что в Vercel Production Branch = `main` и Git-интеграция
  включена; деплои смотри в проекте `fmp-ratings` (§8).
